## 0. 最新推进（2026-03-15）

### 本轮完成

- 阶段 D 已推进到“真实掉落 -> 近距离拾取 -> 携行网格 -> 基地仓储结算”这条闭环，不再只是文本型 loot 记录。
- `RunState` 已接入真实携行网格和地面掉落列表，并完成存档恢复、运行中快照同步和撤离结算回写。
- 战斗中新增 `E` 键拾取；拾取成功后自动放入携行网格，空间不足会明确提示。
- 基地与战区面板都已切到真实网格视图，当前可以直接验证携行占位和仓储占位。
- 基地内的 `储物柜 / 工坊台 / 指挥台` 已接入近距离 `E` 交互，可打开对应上下文面板。
- 基地仓储与战区携行网格都已支持面板内左键拾放与 `R` 旋转，网格开始具备最小可操作性。
- 战区面板现已拆成“附近掉落 + 携行网格”双区，可在面板内把地面物资转入背包，也能把手中物资丢回地面。
- `Tab / I` 面板已改成全屏黑白高对比布局，分区边界、栏位结构和信息层级都比之前更明确。
- 基地仓储与战区携行网格现已支持 `F` 自动整理，会尝试把当前物资重新压缩进可用格位，并兼容手持物品归位。
- 基地仓储与战区携行网格现已改成 `按下拿起 -> 拖动预览 -> 松开放下/回退` 的自由拖拽流程，不再要求二次点击落位。
- 战区携行网格现已支持面板内 `1-4` 快捷栏绑定，绑定关系会随单局快照和存档恢复保留。
- 战区现已补入首批可用消耗品：`治疗针 / 战地急救包 / 震爆罐 / 机动电池`，并开始随敌人掉落进入地面搜刮循环。
- 快捷栏现已支持战斗中 `Z / X / C / V` 实际使用绑定物资，HUD 也会同步显示当前槽位内容与数量。
- 基地 `工坊台` 现已接入真实武器槽位编排，可在近距离交互面板内直接调整 `1 / 2 / 3` 号武器顺序，并回写存档。
- 基地 `指挥台` 现已接入真实路线卡片选择，可在近距离交互面板内切换待部署副本，再前往出击闸门执行部署。
- `npm run build` 已通过。

### 仍未完成

- 快捷栏已接入最小可用闭环，但当前可用物资种类仍偏少，尚未接入更完整的消耗品、制造和战术道具体系。
- 基地近距离交互已接入仓储、武器槽位编排和路线选择，但仍未覆盖维修、模块改装、制造和合同下发等更完整局外流程。
- 地面掉落虽已支持面板内转运与丢回地面，但还没有容器、批量整理和更多物品类型。

## 0.1 视觉与交互统一重构专项（2026-03-15）

### 当前结论

专项实现暂时冻结，先完成 `布局与显示时机设计`。

原因很明确：

- 当前问题已经不是某几个控件不好看，而是整体布局与审美方向不合理
- 如果不先定总布局，继续写代码只会不断返工
- 视觉风格必须严格回到角色、怪物、箭头和子弹特效这套亮色几何语言

专项设计文档见：`visualRefactorPlan.md`

### 先解决的问题

先只回答这些问题，不进入继续编码：

1. 战斗 HUD 的六个固定区块怎么摆
2. 基地常态应该保留哪些常驻信息
3. 哪些内容必须是居中弹出的终端面板，而不是常驻侧栏
4. 小地图、血条、快捷栏、目标信息各自固定在哪
5. 什么提示常驻，什么提示条件出现，什么提示只能居中出现
6. 亮色体系下的边框、角标、箭头、投影和动效如何统一

## 1. 文档目的

本文件用于回答三个问题：

1. 这个项目总共需要开发哪些功能
2. 应该按什么顺序开发
3. 每个阶段需要做哪些重构，以及重构的目标是什么

本文件与 [devDoc.md](d:/shotV/devDoc.md) 的职责不同：

- `devDoc.md` 负责产品与系统设计
- `implementationPlan.md` 负责工程落地与阶段推进

总原则：

`先完成代码功能闭环，再补内容广度`

也就是说，优先让所有系统先“能跑、能存、能组合、能测试”，之后再扩充地图主题、敌人种类、掉落池、事件数量和视觉包装。

## 2. 当前代码基线

当前代码已经具备一个战斗原型基础，主要包括：

- `Vue + PixiJS` 的基础启动与显示层
- 单场景战斗循环
- 玩家移动、射击、冲刺
- 3 把基础武器
- 基础敌人、波次、mini boss
- 命中反馈、屏幕震动、HUD

当前代码的核心问题不是“没有基础”，而是：

- 战斗逻辑主要集中在单个场景文件中
- 缺少“单局游戏状态”的统一定义
- 缺少“基地与长期成长状态”的统一定义
- 缺少存档结构和离线收益结算规则
- 缺少地图、背包、掉落、撤离、资源等模块的清晰边界
- 配置文件尚未完全统一，存在原型阶段遗留结构

这意味着后续开发不能继续在现有战斗场景里堆逻辑，必须先做结构拆分。

## 4. 工程推进原则

### 4.1 顺序原则

功能开发顺序必须遵循：

1. 先拆结构
2. 再定义状态结构和配置结构
3. 再做系统逻辑
4. 再做局内外打通
5. 最后做内容填充

### 4.2 重构原则

每一阶段的重构必须满足：

- 为后续多个系统服务
- 能减少当前场景文件继续膨胀
- 不追求一次性终极架构
- 每次重构后仍然可运行

### 4.3 范围控制原则

MVP 阶段不追求：

- 大量内容
- 复杂联网
- 复杂叙事
- 复杂交易

MVP 阶段追求：

- 全链路可玩
- 结构可扩展
- 数据可持久化
- 系统之间可组合

## 5. 推荐目录重构方向

当前项目建议逐步演进到以下目录结构：

```text
client/src/
  app/
  components/
  game/
    combat/
    core/
    data/
    entities/
    inventory/
    loot/
    meta/
    progression/
    session/
    systems/
    ui/
    world/
```

建议新增模块职责：

- `data/`：所有静态配置
- `session/`：单局游戏中的状态和结算数据
- `world/`：地图、区域、门禁、撤离点
- `inventory/`：背包、仓库、物品摆放逻辑
- `loot/`：掉落生成、拾取和结算
- `meta/`：基地、研究、工坊、离线收益等局外系统
- `progression/`：技能、成就、解锁等长期成长系统
- `systems/`：噪声、耐久、弹种、甲弹对抗等通用系统

## 6. 首轮重构重点

在继续增加新玩法前，必须先做以下结构收口：

### 6.1 统一配置入口

目标：

- 统一敌人、武器、资源、掉落配置入口
- 移除原型阶段重复配置

任务：

- 将旧的 `game/enemies/enemyConfig.ts` 合并或清退
- 统一使用 `data/` 目录管理配置
- 为配置定义稳定的类型接口

### 6.2 拆分战斗场景

目标：

- 把 `CombatSandboxScene.ts` 从“所有逻辑都堆在一起的场景文件”拆成“只负责组织流程的场景入口”

任务：

- 抽离玩家战斗逻辑
- 抽离敌人更新逻辑
- 抽离投射物逻辑
- 抽离 HUD 状态计算
- 抽离波次 / 区域事件逻辑

### 6.3 建立单局游戏状态结构

这里的“单局游戏状态结构”，指的是：

- 玩家从进入地图开始
- 到死亡或撤离结束为止
- 这一整局过程中所有需要记录、结算和保存的状态集合

目标：

- 让一局游戏中的所有关键状态都能被记录、结算和切换

任务：

- 定义单局游戏状态结构
- 定义玩家局内状态
- 定义背包状态
- 定义地图状态
- 定义撤离结算结构

## 7. 分阶段开发计划

以下阶段按“先功能，后内容”的顺序设计。

---

## 阶段 A：架构收口与程序结构分层

### 目标

把当前原型重构成可承载局内外系统的代码基础。

### 本阶段功能

- 场景切换框架
- 基础全局状态管理
- 基础存档结构
- 配置结构统一
- 战斗场景拆分

### 本阶段重构任务

- 将 `CombatSandboxScene.ts` 内的敌人、投射物、HUD、波次逻辑拆到独立模块
- 新增 `session/`、`data/`、`meta/`、`inventory/`、`world/` 基础目录
- 统一静态配置类型
- 增加统一的游戏总状态和存档结构类型：`GameState`、`SaveState`

### 重构目标

- 后续新增背包、地图、掉落时不需要继续向单一场景文件塞逻辑
- 局外与局内能共享存档和配置

### 验收标准

- 当前战斗原型仍可运行
- 代码目录边界清晰
- 配置和状态类型可以稳定复用

---

## 阶段 B：单局游戏状态与结算骨架

### 目标

先把“一局游戏”的数据闭环做出来。

### 本阶段功能

- 一局开始
- 一局结束
- 撤离成功或死亡后的结算
- 局内资源记录
- 局内拾取列表
- 局内临时状态保存

### 本阶段重构任务

- 定义单局状态结构：`RunState`
- 定义玩家单局状态结构：`PlayerRunState`
- 定义撤离或死亡结算结果：`ExtractionResult`
- 定义单件战利品记录：`LootEntry`
- 将当前击杀数、波次、boss 状态接入单局状态结构

### 重构目标

- 以后所有局内系统都基于统一的单局状态结构运行
- 不再依赖散落在场景对象上的临时字段

### 验收标准

- 能开始一局、结束一局、拿到结算结果
- 死亡与撤离可以区分处理

---

## 阶段 C：大地图与区域框架

### 目标

从竞技场原型切到真正的探索地图骨架。

### 本阶段功能

- 程序化大地图生成
- 多区块路线
- 外围区 / 高风险区 / 高价值区占位
- 地图标记与区域状态
- 撤离点占位

### 本阶段重构任务

- 新增 `world/` 模块
- 把当前 arena 边界逻辑替换为地图区块逻辑
- 定义区域、房间、连接关系和门禁接口
- 将敌人生成从“波次场地生成”改为“区域生成”

### 重构目标

- 把游戏的空间结构从“固定战斗场”升级为“可探索战区”

### 验收标准

- 玩家可在一张明显更大的地图中移动
- 地图至少存在多条路径与多个区域标签
- 高风险区和高价值区即使内容简单，也已拥有结构位置

- 护甲耐久

- 新增 `systems/penetration`

## 10. 关键重构目标汇总

整个项目的重构目标可压缩为 6 条：

1. 从单场景原型，重构为局内外双模式项目
2. 从散乱字段状态，重构为统一的单局状态、存档状态、基地状态结构
3. 从固定竞技场，重构为可探索大地图
4. 从纯动作战斗，重构为带资源压力与噪声控制的撤离战斗
5. 从临时拾取反馈，重构为统一物品、背包、仓库系统
6. 从单局爽感原型，重构为带长期成长和离线回流的产品骨架

## 11. 推荐实际开发顺序

如果按最稳妥的工程顺序推进，建议按以下顺序做：

1. 阶段 A：架构收口与程序结构分层
2. 阶段 B：单局游戏状态与结算骨架
3. 阶段 C：大地图与区域框架
4. 阶段 D：掉落、拾取与网格背包
5. 阶段 F：高风险区、钥匙卡、高价值区与撤离
6. 阶段 G：局外基地、工坊、技能与仓库
7. 阶段 E：弹药、耐久、甲弹对抗与噪声
8. 阶段 H：离线收益与后勤闭环
9. 阶段 I：内容填充阶段

说明：

- `阶段 E` 在逻辑上可以早做，但从项目闭环价值来看，放在基地和撤离闭环之后更稳
- 如果中途发现战斗深度明显不足，可将 `阶段 E` 提前到 `阶段 F` 之前

## 12. 最终判断标准

只有当以下条件成立时，才进入内容扩展阶段：

- 局内外切换稳定
- 存档稳定
- 一局流程完整
- 背包与掉落稳定
- 撤离与带出稳定
- 基地成长闭环成立
- 关键资源与损失规则成立

在这之前，不应过早投入大量新敌人、新地图皮肤和新掉落条目。
